# Epic 7: Mise en place de la CI/CD (CI/CD Pipeline Setup)

**Version:** 0.1
**Date :** 2025-05-11
**Statut :** Ébauche

## 1. Objectif de l'Epic

L'objectif de cet Epic est de configurer et de rendre opérationnel le pipeline d'Intégration Continue et de Déploiement Continu (CI/CD) en utilisant GitHub Actions. Ce pipeline automatisera les processus de build, de test, d'analyse de sécurité et de déploiement pour les applications frontend (Astro) et backend (Spring Boot) sur le serveur VPS de production. L'automatisation vise à assurer des livraisons rapides, fiables et sécurisées.

*(Réf PRD : "Epic 7: Mise en place de la CI/CD (CI/CD Pipeline Setup) - Goal: Configurer le pipeline de CI/CD avec GitHub Actions pour automatiser les builds (frontend Astro, backend Spring Boot), les tests (Vitest, Cypress, tests backend), l'analyse de sécurité (Trivy) et les déploiements sur le VPS via Docker Compose.")*
*(Réf principal : `docs/ci-cd/pipeline.md`)*

## 3. Critères d'Acceptation Généraux pour l'Epic

* Les workflows GitHub Actions pour le frontend et le backend sont créés, fonctionnels et déclenchés correctement par les `push` et `pull_request` sur les branches et chemins configurés.
* Les étapes de linting, tests unitaires/intégration, build applicatif, et scans de sécurité des dépendances sont exécutées avec succès pour les deux applications.
* Les images Docker pour le frontend et le backend sont construites, scannées avec Trivy, et poussées vers GHCR avec des tags appropriés lors des `push` sur `main` et `develop`.
* Le déploiement automatisé sur le VPS de production via SSH est fonctionnel pour les deux applications lors d'un `push` sur la branche `main`.
* Les migrations de base de données Liquibase sont appliquées automatiquement lors du déploiement du backend.
* Les échecs dans les étapes critiques (tests, scans de sécurité, build) entraînent l'échec du workflow et empêchent le déploiement.
* La documentation `docs/ci-cd/pipeline.md` est complète et reflète l'implémentation.

## 4. Dépendances

* **Epic 1 :** VPS configuré, Docker/Docker Compose installés, utilisateur de déploiement SSH, fichiers `docker-compose.prod.yml` et `.env` de production en place sur le VPS. Accès à GHCR.
* `Dockerfile` pour le frontend et le backend prêts et fonctionnels.
* Scripts de test (`pnpm test:unit`, `mvnw test`, etc.) définis dans les `package.json` et `pom.xml`.
* Configuration des outils de scan (Trivy, OWASP Dependency-Check, PNPM Audit) dans les workflows.

## 5. Risques et Questions Ouvertes

* **Complexité des workflows YAML GitHub Actions :** Peut nécessiter des itérations pour la mise au point.
* **Gestion des permissions pour la clé SSH sur le VPS :** S'assurer que l'utilisateur de déploiement a juste les droits nécessaires.
* **Temps d'exécution des pipelines :** Optimiser les caches et paralléliser les jobs pour maintenir des temps de feedback raisonnables, surtout pour les PRs.
* **Fiabilité des tests E2E :** Les tests E2E peuvent être instables ("flaky"). Une stratégie de re-essai ou une exécution sur un environnement de staging dédié peut être nécessaire post-MVP.

## 6. Hors Scope pour cet Epic

* Configuration d'un environnement de staging dédié et des déploiements automatisés vers celui-ci.
* Stratégies de déploiement avancées (Blue/Green, Canary).
* Notifications de pipeline avancées (Slack, Teams).
* Optimisation poussée des temps de build au-delà du caching de base.
