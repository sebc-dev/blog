# Epic 4: Interaction Utilisateur Minimale (MVP User Engagement)

**Version:** 0.1
**Date :** 2025-05-11
**Statut :** Ébauche

## 1. Objectif de l'Epic

L'objectif de cet Epic est de mettre en œuvre les fonctionnalités minimales d'engagement utilisateur pour le MVP. Cela comprend la capacité pour les utilisateurs de partager des articles sur les réseaux sociaux et de fournir un feedback simple sur l'utilité de chaque article. Ces interactions seront comptabilisées de manière anonyme par le backend Spring Boot pour fournir des métriques de base sur l'engagement du contenu.

*(Réf PRD : "Epic 4: Interaction Utilisateur Minimale (MVP User Engagement) - Goal: Mettre en place les systèmes de partage sur les réseaux sociaux (frontend avec appel API backend pour comptage anonyme) et le module de feedback "Cet article a été utile ? Oui/Non" (frontend avec appel API backend pour comptage anonyme). Le backend Spring Boot gérera le stockage de ces compteurs. (Les commentaires sont reportés post-MVP).")*

## 3. Critères d'Acceptation Généraux pour l'Epic

* Les endpoints backend pour `/share` et `/feedback` sont fonctionnels, testés, et incrémentent/créent correctement les enregistrements dans la table `article_version_metrics`.
* Les boutons de partage social sur les pages d'articles ouvrent les dialogues de partage des plateformes respectives ET déclenchent l'appel de comptage au backend.
* Le module de feedback "Article utile ?" est fonctionnel sur les pages d'articles, envoie le vote au backend, et affiche un message de remerciement.
* Toutes les interactions de comptage sont anonymes (aucune information utilisateur n'est envoyée au backend à part l'`articleCanonicalSlug` et la `lang`).
* Les appels API depuis le frontend vers le backend pour ces fonctionnalités sont asynchrones et gèrent les erreurs de manière à ne pas impacter négativement l'UX principale.
* L'UI des composants de partage et de feedback est conforme aux spécifications de `docs/ui-ux/ui-ux-spec.md`.

## 4. Dépendances

* **Epic 1 :** Infrastructure backend (Spring Boot, PostgreSQL) et frontend (Astro) en place.
* **Epic 3 :** Pages d'articles disponibles pour y intégrer les composants d'interaction.
* Documents de référence : `docs/architecture/api-reference.md`, `docs/data-models.md`, `docs/ui-ux/ui-ux-spec.md`.

## 5. Risques et Questions Ouvertes

* **Prévention du Spam/Abus des API de comptage :** Pour le MVP, les endpoints sont publics. Le rate limiting est une considération post-MVP. Quel niveau d'abus est acceptable pour le MVP ? (Probablement faible risque initialement).
* **Gestion de l'état "voté" pour le feedback :** La solution `sessionStorage` est simple pour le MVP. Des solutions plus persistantes (ex: `localStorage` avec une portée plus longue, ou un système de vote basé sur l'IP avec anonymisation si la RGPD le permet et si c'est jugé nécessaire) sont post-MVP.
* **Design exact des messages de remerciement/confirmation.**

## 6. Hors Scope pour cet Epic

* Système de commentaires complet.
* Interactions utilisateur plus avancées (sauvegarde d'articles, notations complexes, etc.).
* Affichage des compteurs de partage ou d'utilité sur le frontend (pourrait être une Story dans un Epic futur si souhaité).
